### SLIDE 5-6 | AI 에이전트란? — 챗봇 vs 에이전트 비교

**청중이 가장 헷갈려하는 개념 → 초반에 명확히 정의 필요**

| 구분 | 일반 AI 챗봇 | AI 에이전트 |
|------|-------------|------------|
| 동작 방식 | 질문하면 답변 (1회성) | 목표를 받고 스스로 계획 → 실행 → 검토 |
| 도구 사용 | 텍스트만 생성 | 이메일 발송, 파일 검색, 코드 실행, 웹 접속 등 |
| 자율성 | 사람이 매번 지시 | 단계별로 스스로 판단 후 실행 |
| 예시 | "이메일 초안 써줘" | "이번 주 미팅 일정 보고 참석자에게 알림 자동 발송해줘" |

**핵심 비유 (발표에서 활용):**
> "챗봇은 질문에 답하는 비서고, AI 에이전트는 스스로 판단하고 실행하는 직원이다."
> OpenClaw 사용자 후기: "WhatsApp으로 문자 보내면 컴퓨터가 알아서 한다. 마치 항상 책상에 앉아있는 직원 같다."

#### SLIDE 9 | OpenClaw의 광폭행보

**OpenClaw(구 Clawdbot)란?**
- 오스트리아 개발자 Peter Steinberger가 **2025년 11월** 주말 프로젝트로 시작
- 기존 챗봇과 근본적으로 다른 접근: **LLM에 컴퓨터를 통째로 연결**
- WhatsApp, Telegram, Discord, Slack으로 문자 보내면 → 컴퓨터가 실제로 작업 실행

**주요 기능:**
- 파일 읽기/쓰기, 쉘 명령어 실행, 웹 브라우징
- 이메일 발송, 캘린더 관리, API 제어
- **100개 이상의 "AgentSkills"** — 커뮤니티 사용자가 계속 추가 중
- 24/7 자율 실행 (cron job으로 스케줄 설정)

**폭발적 성장:**
- **60일 만에 GitHub stars 250,000+** → GitHub 역사상 가장 빠른 성장 프로젝트 중 하나
- React(페이스북 핵심 라이브러리)가 10년 넘게 쌓은 star 수를 2개월 만에 돌파
- **하루 만에 25,000 stars** 증가 기록
- 사용자 반응: "이건 Jarvis다. 이미 존재한다." / "Apple이 수십 년간 못 만든 걸 오픈소스가 만들었다."

**왜 중요한가:**
- 개인 에이전트(Personal Agent) 시대의 신호탄
- 오픈소스라서 누구나 무료로 사용 가능 (API 비용만 부담)
- 개발자가 아니어도 WhatsApp으로 업무 자동화 가능

### SLIDE 14-15 | VBA 소개 + 실무 자동화 예시

#### SLIDE 14 | VBA란 무엇인가?

**VBA (Visual Basic for Applications):**
- Microsoft 365 전 제품(Excel, Word, PowerPoint, Outlook)에 내장된 자동화 언어
- "엑셀에서 내가 손으로 하는 모든 작업을 기록하고 반복 실행하는 도구"
- AI(GPT/Claude) + VBA = 코딩 몰라도 자동화 가능

**왜 VBA인가? (2026년에도 여전히 강력한 이유)**
- 추가 설치 없음 — 이미 Excel에 다 있음
- 회사 IT 정책 승인 불필요
- Excel, Word, PPT, Outlook 모두 연동
- GPT가 코드를 전부 생성이 가능

#### SLIDE 15 | VBA로 자동화 가능한 실무 작업 6가지

| # | 작업 유형 | 구체적 예시 | 예상 효과 |
|---|-----------|-------------|---------|
| 1 | **정기 보고서 자동 생성** | 월말 매출 데이터 → 정렬 → 차트 생성 → PDF 저장까지 원클릭 | 2시간 → 3분 |
| 2 | **이메일 자동 발송** | 마감 알림, 미지급 거래처 안내 메일 (Outlook 연동) | 반복 업무 제거 |
| 3 | **인보이스/정산서 자동화** | 데이터 입력 → 양식 자동 생성 → 파일명 저장 | 오류 최소화 |
| 4 | **데이터 정제 & 통합** | 여러 시트 데이터 합치기 + 중복 제거 + 포맷 통일 | 수작업 제거 |
| 5 | **마감 체크리스트** | 조건부 서식 + 항목별 마감 자동 알림 | 누락 방지 |
| 6 | **PPT/Word 자동 업데이트** | Excel 데이터 변경 시 보고서 자동 반영 | 이중 작업 제거 |

### SLIDE 16-17 | GPT+VBA 실전 방법론

#### SLIDE 16 | 모듈화 접근법 — 자동차 비유

**핵심 개념: 한 번에 전부 만들려 하지 않는다**

```
잘못된 방법:
"인보이스 전체 자동화 코드 만들어줘" → GPT가 복잡한 코드 → 오류 → 막힘

올바른 방법 (모듈화):
엔진 모듈:  "파일 열고 A열 데이터 읽는 코드"
변속기 모듈: "날짜 형식 통일하는 코드"
차체 모듈:  "결과를 새 시트에 정리하는 코드"
완성차:    "위 3가지를 순서대로 실행하는 코드"
```

**실전 4단계:**
```
Step 1. 업무 모듈화
   지금 직접 엑셀로 하는 작업을 단계별로 적기
   예) 파일 열기 → 데이터 복사 → 정렬 → 합산 → 이메일 발송

Step 2. 단계별 GPT 요청 (한 단계씩!)
   "엑셀 A열 데이터를 Sheet2로 복사하는 VBA 코드 만들어줘"

Step 3. 각 모듈 독립 테스트
   백업 파일에서 먼저 실행해보기

Step 4. 통합 VBA 코드 완성
   모든 모듈을 순서대로 호출하는 메인 코드 작성
```

### SLIDE 18 | 프롬프트 작성 원칙 3가지

**GPT에게 "잘" 요청하는 법 (VBA 포함 모든 AI 활용 시 공통 적용)**

**원칙 1. 역할을 부여하라**
```
❌ "VBA 코드 만들어줘"
✅ "너는 10년 경력의 엑셀 VBA 전문가야. 다음 작업을 자동화하는 코드를 만들어줘."
```

**원칙 2. 맥락(Context)을 제공하라**
```
❌ "정산서 만들어줘"
✅ "Sheet1의 A열에 날짜, B열에 거래처명, C열에 금액이 있어.
    이 데이터를 Sheet2에 고객사별로 합산해서 정리하는 VBA 코드 만들어줘."
```

**원칙 3. 출력 형식을 지정하라**
```
❌ "설명해줘"
✅ "코드와 함께 각 줄이 무엇을 하는지 한국어 주석도 달아줘.
    그리고 테스트 시 주의사항도 알려줘."
```

> 핵심: 프롬프트 = AI에 대한 업무 지시서. 신입직원에게 업무를 지시한다고 생각하고 구체적으로 작성할수록 좋은 결과가 나온다.

### SLIDE 19 | GPT+VBA 코딩 시 유의사항 6가지

**① 테스트 데이터로 먼저 실행**
- 실제 데이터에 바로 적용하면 데이터 손실 위험
- 항상 백업 파일 또는 더미 데이터로 먼저 테스트

**② AI는 내 파일 구조를 모른다**
- 시트명, 열 위치, 데이터 범위를 반드시 명시
- "Sheet1의 A2:A100에 거래처명이 있고..." 와 같이 구체적으로

**③ 오류 메시지는 그대로 GPT에게 붙여넣기**
- VBA 오류 발생 시 → 오류 번호 + 메시지 복사 → GPT에 그대로 붙여넣기
- GPT가 즉시 원인과 수정 코드를 제공

**④ 한 번에 너무 많이 요청하지 말 것**
- 긴 코드보다 짧은 모듈을 차례대로 만드는 것이 훨씬 안정적

**⑤ 코드를 이해하려는 노력 (포기하지 말 것)**
- "이 코드가 무엇을 하는지 한줄씩 설명해줘" 요청 습관화
- 이해 없이 붙여넣기만 하면 오류 시 속수무책

### SLIDE 20-21 | 미래 업무환경 & 시사점 (오케스트레이터 프레임)

#### SLIDE 20 | 업무 환경의 진화 & 3가지 시사점

**역사적 흐름:**
```
20년 전:  터미널 → GUI (마우스, 창) → 엑셀/워드/PPT → 이메일/인터넷
지금:     AI 에이전트의 등장
```
> 매번 새로운 도구가 등장할 때마다, 그것을 먼저 잘 쓰는 사람이 경쟁력을 가졌다.

**시사점 1. 아이디어만 있으면 누구나 만들 수 있다**
> 아이디어 → AI 에이전트 → 프로덕트/서비스
- 에이전트 구축 시간: **15~60분** (노코드 플랫폼)
- 2026년까지 기업 소프트웨어의 약 **40%가 자연어 기반으로 개발** 전망
- 해커톤 1위: 변호사가 1주일 만에 건축 허가 자동화 툴 완성

**시사점 2. 도메인 전문성이 새로운 경쟁력**
> 예전: 기술을 가진 사람이 문제를 해결했다
> 지금: 문제를 가진 사람이 기술을 쓰는 시대
- IBM 2026: "AI 에이전트 창조 능력이 개발자를 넘어 일반 비즈니스 사용자로 이동하고 있다"
- 진짜 문제에 가장 가까운 사람이 혁신을 주도하게 된다

**시사점 3. 오케스트레이터로의 전환**
> 우리는 이제 기능적인 일을 직접 하는 것이 아니라,
> 여러 AI 에이전트들을 지휘하는 역할로 변하고 있다

```
과거: 혼자 바이올린을 연주
현재: 오케스트라 지휘자
```

- Salesforce: 직원이 AI 에이전트를 감독하는 새로운 직무 정의
- Morgan Stanley: "AI 덕분에 더 적은 인원으로 더 많은 성과를 내는 기업이 늘고 있다"
- C.H. Robinson: "미래의 직무는 AI 에이전트를 관리하고 컨텍스트를 제공하는 것"
- **미래 경쟁력: AI 에이전트를 복합적으로 사용하고 효율을 극대화하는 사람**

#### SLIDE 21 | 프롬프트 → 컨텍스트 → 에이전틱 엔지니어링

**AI 활용 역량의 3단계 진화:**

```
1단계: 프롬프트 엔지니어링
       "어떻게 질문할까?" — AI에게 잘 물어보는 기술

             ↓

2단계: 컨텍스트 엔지니어링
       "무엇을 알려줄까?" — AI가 올바른 답을 내도록
       적절한 정보를 제공하는 기술
       (Anthropic 공식 블로그, 2025.09 정식 정의)

             ↓

3단계: 에이전틱 엔지니어링
       "어떤 AI 팀을 어떻게 조율할까?"
       여러 에이전트를 설계하고 지휘하는 기술
```

**Andrej Karpathy (OpenAI 공동창업자) 정의:**
> "Context engineering is the delicate art and science of filling the context window with just the right information for the next step."
> (컨텍스트 엔지니어링이란, 다음 단계를 위해 딱 필요한 정보로 AI의 맥락을 채우는 섬세한 기술이다.)

**결론:** 우리는 점점 AI 에이전트들의 오케스트레이터로서의 역량을 키워야 한다.
